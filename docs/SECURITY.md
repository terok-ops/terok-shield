# Security Model

## Threat model

terok-shield protects against a **compromised container workload** that attempts to:

1. **Exfiltrate data** to attacker-controlled servers
2. **Attack the LAN** (lateral movement to host or other machines on RFC1918 networks)
3. **Download malware** from arbitrary internet hosts
4. **Connect to a C2 server** for remote control

The attacker has **arbitrary code execution inside the container** but does not have root on the host. The container runs rootless under Podman.

### Out of scope

- Host kernel exploits (container escape)
- DNS exfiltration through allowed resolvers
- Covert channels via allowed destinations
- Attacks against the gate server itself (separate component)

The workload cannot modify the nftables rules because:

- `CAP_NET_ADMIN` is dropped вҖ” `nft` commands fail with EPERM
- `CAP_NET_RAW` is dropped вҖ” no raw socket access
- `no-new-privileges` is set вҖ” cannot regain capabilities

### nft.py вҖ” the auditable security boundary

All nftables rulesets are generated by a single file: `nft.py`. This file has **zero non-stdlib imports** (only `ipaddress`, `re`, `textwrap`) so it can be audited in isolation without understanding the rest of the codebase.

Properties enforced by design:

- **Input validation before interpolation** вҖ” `safe_ip()` validates all IP addresses/CIDRs through `ipaddress` module before they enter any nft command string. `safe_name()` validates identifiers against `^[a-zA-Z0-9_-]+$`.
- **RFC1918 blocks before allow-set** вҖ” the nftables chain evaluates RFC1918 reject rules structurally before the allow-set check. A bug in allowlist management cannot accidentally allow RFC1918 traffic.
- **Self-verification** вҖ” `verify_ruleset()` checks post-apply invariants: policy is drop, reject type present, deny log prefix present, all RFC1918 ranges blocked.
- **Import isolation** вҖ” enforced by an AST test that parses `nft.py` and asserts only stdlib modules are imported, plus bandit SAST scanning.

### Chain evaluation order

**Standard mode** (per-container netns, output chain):

```
loopback вҶ’ established вҶ’ DNS вҶ’ gate port вҶ’ RFC1918 reject вҶ’ allow_v4 вҶ’ deny all
```

**Hardened mode** (rootless-netns, forward chain):

```
established вҶ’ ICMP вҶ’ DNS вҶ’ gate вҶ’ RFC1918 reject вҶ’ intra-bridge вҶ’ allow_v4 вҶ’ deny all
```

## Fail-closed guarantees

After running `terok-shield setup`, the following invariant holds: **no path from "firewall setup failed" to "container running unrestricted."**

| Failure | Result |
|---------|--------|
| OCI hook raises an exception | Container torn down by podman (non-zero exit) |
| `nft` binary missing | `ExecError` вҶ’ hook exits non-zero вҶ’ container torn down |
| Profile not found | `FileNotFoundError` вҶ’ hook exits вҶ’ container torn down |
| Ruleset fails to load | Hook exits вҶ’ container torn down |
| Self-verification fails | Hook exits вҶ’ container torn down |

### Preconditions

Before `terok-shield setup` is run, `is_shield_active()` returns `False` and containers start **without** firewall rules (a warning is printed). The fail-closed guarantee only applies after setup completes successfully.

## DNS allowlisting

Allowed destinations are specified as **domain names** in `.txt` files, not hardcoded IP ranges. This avoids staleness when CDN providers rotate IP addresses.

Resolution flow:

1. `terok-shield setup` / `terok-shield resolve` resolves all domains via `dig +short A`
2. Results cached with timestamp in `state_dir/resolved/`
3. On container start, cache is used if fresh (< 1 hour); otherwise re-resolved
4. OCI hook reads cached IPs and adds them to the `allow_v4` nft set

### Allowlist profiles

Profiles are collections of domain names grouped by use case:

- `base.txt` вҖ” OS package repos (Ubuntu archive/security)
- `dev-standard.txt` вҖ” GitHub, PyPI, npm, Docker Hub
- `dev-python.txt` вҖ” PyPI, conda-forge
- `dev-node.txt` вҖ” npm, yarn registries
- `nvidia-hpc.txt` вҖ” NVIDIA downloads, NGC

## Audit logging

All connection events are logged as JSON-lines to `state_dir/logs/{container}.jsonl`:

```json
{"ts":"2026-03-07T12:00:00+00:00","container":"myproj-cli-1","action":"setup","detail":"profiles=dev-standard"}
{"ts":"2026-03-07T12:00:01+00:00","container":"myproj-cli-1","action":"allowed","dest":"140.82.112.4","detail":"domain=github.com"}
```

In-kernel audit is also enabled via nftables log rules:

- `TEROK_SHIELD_DENIED:` вҖ” logged for all rejected packets
- `TEROK_SHIELD_ALLOWED:` вҖ” logged for allowed packets (rate-limited to 10/second)
